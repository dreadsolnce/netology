### Основные компоненты API Gateway
- Внешний интерфейс (API Facade)
- Маршрутизатор запросов (Request Router)
- Менеджер аутентификации и авторизации
- Балансировщик нагрузки
- Кэш-слой
- Трансформатор запросов/ответов
- Мониторинг и логирование
- Ограничитель запросов (Rate Limiter)
### Среди популярных решений для реализации API Gateway можно выделить:
- Kong — производительный Gateway на базе NGINX с открытым исходным кодом,
- Spring Cloud Gateway — часть экосистемы Spring Cloud для Java-приложений,
- Amazon API Gateway — полностью управляемый сервис от AWS,
- Azure API Management — аналогичное решение от Microsoft,
- Express Gateway — легковесное решение на Node.is

# Express Gateway
 - Базовая маршрутизация
 - Аутентификация
 - Агрегация данных
 - Кэширование
 - Circuit Breaker для защиты от каскадных отказов.
 - Логирование

# Kong
- проксирование трафика и балансировка нагрузки
- соблюдение правил безопасности (например, проверка JWT)
- ограничение запросов и ограничение количества запросов;
- выполнение плагинов и middleware

##### Мой выбор: Kong
Удовлетворяет требованиям описанным в задании, а так же имеет множество плагинов расширения и активное коммьюните. Есть бесплатная версия.

### Брокеры сообщений

#### Apache Kafka
- Подходит для обработки больших объёмов данных в реальном времени.

Принцип работы: отправители формируют сообщения и передают их брокеру, программа раскладывает полученные сообщения по отдельным топикам (topics) в соответствии с потребностями получателей, а затем получатели забирают необходимые данные.

Преимущества:
- масштабируемость
- отказоустойчивость
- возможность хранения данных

#### RabbitMQ
- Подходит для систем, где важны надёжность доставки сообщений, сложная маршрутизация и управление очередями, но объём данных не столь велик. 

Преимущества:
- Гибкая маршрутизация — сообщения маршрутизируются через обменник (exchange) перед попаданием в очереди.
- Подход к обмену сообщениями — брокер сам активно отправляет сообщения потребителям, которые подписаны на очереди.
- Удаление сообщений из очереди — после получения потребителями сообщение удаляется из очереди, что позволяет одно и то же сообщение обработать только одним потребителем.

#### Redis
- Хранит данные в оперативной памяти, что обеспечивает высокую скорость работы и низкую задержку.
- Ограниченная надёжность — данные по умолчанию хранятся только в RAM, и без настроенной репликации или снапшотов могут быть потеряны при сбое

Преимущества:
- Поддержка обмена сообщениями через механизм Pub/Sub (Publish/Subscribe).
- Сверхнизкая задержка — задержки измеряются микросекундами.
- Широкие возможности кеширования — быстрый доступ к часто запрашиваемым данным, хранение сессий, счётчиков, рейтингов.
